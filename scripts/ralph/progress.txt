## Codebase Patterns
- next-intl v4.8.3 is used with Next.js 15; plugin configured via `createNextIntlPlugin("./i18n/request.ts")` in `next.config.ts`
- Translation messages live in `i18n/messages/{locale}.json`; imported dynamically in `i18n/request.ts`
- `i18n/routing.ts` defines locales `['zh', 'en']` with `defaultLocale: 'zh'`
- For static export, `setRequestLocale(locale)` must be called in every server component that uses translations
- `generateStaticParams` must return `[{ locale: 'zh' }, { locale: 'en' }]` for `[locale]` segment
- Translation keys are organized by section: `site.*`, `nav.*`, `hero.*`, `now.*`, `work.*`, `papers.*`, `press.*`, `awards.*`, `about.*`, `footer.*`
- `i18n/navigation.ts` exports locale-aware `Link`, `usePathname`, `useRouter`, `redirect` via `createNavigation(routing)`
- Components using `Link` should import from `@/i18n/navigation` (not `next/link`) for automatic locale prefixing
- `app/layout.tsx` is a bare pass-through (returns just children); the `app/[locale]/layout.tsx` provides `<html>`, `<body>`, fonts, and `NextIntlClientProvider`
- `useRouter().push(path, { locale })` from `@/i18n/navigation` switches locales; `usePathname()` returns path without locale prefix
- `generateMetadata` in `app/[locale]/layout.tsx` uses `getTranslations({ locale, namespace })` for locale-aware SEO metadata
